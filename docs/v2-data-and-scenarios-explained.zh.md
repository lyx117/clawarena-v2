# ClawArena-V2 数据与场景概览（中文版）


## ClawArena-V2 是什么

ClawArena-V2 是一个面向命令行智能体的交互式 benchmark。它评测的不是“模型能不能给出一个看起来合理的最终答案”，而是“智能体能不能在一个持续变化的环境里，一步一步完成任务”。

更具体一点，V2 关心的是这样一种能力：

- 智能体先读懂任务 instruction
- 再查看当前环境里已经存在什么
- 然后逐步发出命令
- 每执行一步，环境状态都会变化
- 智能体还要根据新的 observation 继续往下做
- 最后真的把 workflow 收尾，而不是只说出一个计划

所以，V2 不是传统的 one-shot 文本题，也不是只考“会不会描述正确策略”的 benchmark。它更像是在问：智能体能不能在真实的、会变化的软件状态上把事情做成。

## 为什么要做这种 benchmark

很多 agent benchmark 最大的问题不是“想法不对”，而是构造成本太高。真正像样的交互任务，如果全靠人工一个个写，通常会遇到这些麻烦：

- 任务 instruction 本身要写得合理
- 初始状态要配得对，不能自相矛盾
- 参考 workflow 要讲得清楚
- 成功条件要能落成可执行检查，而不是靠人眼看
- 任务一多以后，维护、修订、扩展都很痛苦

ClawArena-V2 的一个核心思路就是：不要把 benchmark 理解成“手写一张题表”，而要把它理解成“从可复用场景族自动生成 executable tasks 的系统”。

换句话说，V2 不是先有一堆固定题目，再拿环境去跑；而是先定义一批可复用的 workflow semantics，然后用生成器把它们编译成具体任务。

## 数据是怎么造出来的

V2 的数据不是一份人工手写、固定不变的题目清单。它是从可复用的场景族自动生成出来的。

一个场景族可以理解成一种反复出现的 workflow 模式。比如：

- 事情其实已经做完了，所以智能体不应该重复做
- 整个 workflow 只缺了一步，所以智能体应该只补缺口
- 当前状态是错的或者过期了，所以智能体应该修复或替换
- 多个信息源互相冲突，所以智能体必须选对分支
- 一个局部步骤虽然做对了，但整个 workflow 还没有真正收尾

对每个生成出来的任务，流水线通常会组合几类东西：

- 一个场景模板
- 具体落地后的变量和值
- 一个初始环境状态
- 一个预期的 workflow 结构
- 一个可执行的 validator
- 用于统计、分组和报告的元数据

所以，一个生成任务不只是一个 instruction 字符串。它通常至少还包含：

- 任务要在什么初始状态下开始
- 这个任务属于哪个场景族
- 成功时环境应该变成什么样
- 应该检查哪些结果和副作用
- 这个任务在 benchmark 统计里该如何归类

这也是为什么 V2 的任务更像“可执行 benchmark specification”，而不是“只给人看的题目描述”。

## 一个生成任务里通常有什么

如果把一个 V2 任务拆开看，里面一般会有几层信息。

第一层是**表面上给智能体看的部分**。这包括：

- 任务 instruction
- 智能体在当前时刻可见的环境状态

第二层是**生成器在背后补齐的结构**。这包括：

- 这个任务属于哪个 scenario family
- 这个 family 的哪些变量被具体实例化了
- 当前任务用了什么名字、时间、对象和冲突条件

第三层是**评测用的信息**。这包括：

- 参考 workflow 结构
- 可执行 checks
- 报告和分组用的 metadata

这三层放在一起，才构成了一个真正可运行、可评测、可复现的 benchmark task。

## 场景是怎么造出来的

每个场景族都对应一种反复出现的 workflow 模式或失败模式。生成器再通过改变初始状态和任务细节，从这个场景族实例化出很多具体任务。

用更直白的话说，生成器会保留“同一种核心局面”，但把具体实例换掉。一个任务可能涉及某条 inbox 线程、某个 calendar item 和某个 deadline。另一个来自同一家族的任务，保留同样的模式，但会把名字、时间、缺失步骤或者冲突证据换成另一组。

这就是 benchmark 能扩展的原因。它不是每次都从头手写一批新题，而是复用同一套 family semantics，再实例化出很多具体 rollout。

下面是几个典型例子。

### 已完成 / 避免重复

这类场景是在检查：所需工作是不是已经存在。

智能体应该先查看环境，发现任务已经完成，或者已经被环境中的对象正确表示出来，然后避免再创建一份重复对象。这类场景很有用，因为很多智能体会过度积极地“动手”，即使正确做法其实是什么都别多做。

### 部分状态 / 补全缺口

这类场景的重点是找到 workflow 里缺的那一块。

比如任务已经有了，但 calendar event 还没有；或者草稿文件已经存在，但后续动作还没做。智能体应该只补上缺失的部分，而不是把整条 workflow 全部重建一遍。

### 过期或错误状态 / 修复或替换

这类场景要求智能体意识到：当前状态不只是“不完整”，而是“真的有问题”。

有时正确做法是修复现有对象；有时正确做法是把它替换成一个正确的新对象。这类场景难在，智能体必须分清楚“缺失了”“已经正确了”“当前是错的”这三种不同情况。

### 冲突证据 / 分支选择

这类场景是在测试：智能体能不能在行动前综合多个信号。

例如，一封邮件说的是一件事，但 calendar entry 或另一个来源暗示的是另一件事。智能体必须解决冲突并选择正确分支，而不是盲目跟随它看到的第一条证据。

### 跨 tasks、calendar、email、files 的 workflow 收尾

这类场景是在测试：智能体能不能把整个 workflow 真正收完，而不是只做对某一个局部步骤。

有些智能体会正确更新一个 task，但忘了后续文件、calendar event 或沟通步骤。这类场景检查的是，智能体能不能跨多个 surface 把整个 workflow 一路做完。

## 为什么这样生成是有价值的

这种生成式构造有几个很实际的好处。

第一，它比纯人工手写更容易扩展。因为你定义的是 family semantics，而不是一题一题硬写。

第二，它更容易维护。一个 family 的逻辑修好以后，整批任务都可以一起刷新，而不是到处手动修补。

第三，它更容易审计。因为任务不只是自然语言描述，还带着初始状态、validator 和 metadata，别人更容易看清楚任务到底在测什么。

第四，它更容易控制 release profile。你可以调每个场景族生成多少条任务，可以重新平衡 benchmark 难度和覆盖面，而不是被一份旧题库锁死。

## 这个 benchmark 为什么难

这个 benchmark 难，是因为 workflow 本身就难，不是因为 checker 故意“下套”。

真正难的点，都是现实智能体经常出问题的地方：

- 正确状态有一部分已经存在
- 智能体必须避免重复已有的有效工作
- 下一步取决于是否选对分支
- 错误状态必须被修复，而不是被忽略
- 过期状态必须被替换，但又不能破坏已有的正确状态
- 成功意味着把整个 workflow 做完，而不是只完成其中一个小步骤

这也是为什么 V2 比 one-shot benchmark 更有信息量。它能区分“智能体会不会谈论 workflow”和“智能体能不能真的把 workflow 做完”。

更重要的是，这种难度来自 workflow semantics 本身，而不是来自一些人为设计的怪规则。也就是说，benchmark 的难点主要来自：

- 对已有状态的判断
- 对分支的选择
- 对修复和替换的区分
- 对整条 workflow 是否收尾的把握

这类难点是 agent 真正会在实际系统里遇到的。

## 智能体在 V2 里是怎么跑起来的

在 V2 里，一个 rollout 不是“给一个 prompt，回一个答案”这么简单。更接近的流程是：

1. 给智能体一个任务 instruction
2. 智能体查看当前 observation 和可见环境状态
3. 智能体输出一条命令
4. 环境执行这条命令
5. 环境返回新的 observation 和更新后的状态
6. 智能体继续下一步，直到成功或者步数耗尽

也就是说，V2 的评测单位本质上是一个多步 rollout，而不是单轮问答。

这也是为什么很多静态 benchmark 里看不出来的问题，在 V2 里会暴露出来，比如：

- 明明已经做对了还重复创建对象
- 只做完局部步骤，没有收尾
- 读到一条证据就草率行动，没处理冲突信息
- 没分清“缺失”还是“错误”

## 评测在大白话里是怎么做的

一次 rollout 结束后，benchmark 不会只看智能体最后吐出的那段文本。它会构造一个标准化的 evaluation state，把发生过的事情用一致的方式整理出来。

简单说，这个标准化状态会包括：

- 最近的命令历史
- 命令输出
- 当前 backend state
- 各种 effect traces，比如创建了任务、更新了事件、发出了邮件、修改了文件

然后 evaluator 会检查：环境里到底变成了什么样，以及有哪些可观察的副作用真的发生了。

这里的关键点是：评测关注的是**结果和副作用**，而不是某一条唯一正确的命令序列。

这也是为什么多条不同 trajectory 依然可能都算通过。如果两个智能体走了不同的命令序列，但最后落在同一个正确的 evaluated outcome 上，它们都可以算成功。这个 benchmark 是 result-first，而不是 exact-trajectory-first。

## full-pass accuracy 和 average score 有什么区别

V2 里通常会同时看两类指标。

第一类是 **full-pass accuracy**。它问的是：这个任务是不是完整满足了所有必需检查。这个指标更严格，适合看“任务到底有没有真正完成”。

第二类是 **average score**。它允许部分分。如果智能体做对了很多步骤，但最后没有彻底收尾，或者漏掉了一两个必要动作，它仍然能拿到一定分数。

这两个指标一起看会更有信息量，因为很多 agent 的真实问题不是“完全不会做”，而是“做到一半，或者只差最后收尾一步”。

## 当前官方 profile 怎么理解

当前官方的 V2 hard profile 是：

- `17` 个 hard scenarios
- `362` 个 hard tasks
- `1616` 个 total tasks

这些数字描述的是当前官方发布 snapshot，不是 benchmark family 的硬上限。

换句话说，这些数字表示的是“当前官方版本发布出来有多少任务”，而不是说 V2 天生就只能有这么多。

当前最核心的 benchmark family 是：

- `hard_decision_workflow`

当前 hard suite 里最主要的六个 reporting buckets 是：

- `duplicate_avoidance`
- `gap_completion`
- `information_transfer`
- `multi_source_reasoning`
- `state_repair`
- `workflow_completion`

这些 bucket 可以理解为：我们在看模型结果时，最常用来分解能力差异的几个主维度。

## 这份 benchmark 最想证明什么

如果把 V2 的目标用一句话说清楚，那就是：

> 我们想要的不只是一个“看起来像任务”的题库，而是一套可以自动生成、可以真实执行、也可以用结果导向方式稳定评测的交互式 benchmark。

也正因为如此，V2 的价值不只是“题目更多”，而是：

- 任务可以自动扩展
- 场景可以系统化地覆盖
- rollout 可以真实执行
- evaluator 可以不依赖唯一轨迹
- benchmark 可以真正测出不同模型在 workflow 上的差异
